                            1. EXECUTIVE SUMMARY
================================================================================

This graduation project presents a comprehensive AI-powered Decision Support
System designed specifically for the Egyptian criminal justice system. The
system leverages cutting-edge technologies including:

• Multi-Agent Architecture for specialized legal tasks
• Graph RAG (Retrieval-Augmented Generation) for knowledge retrieval
• Neo4j knowledge graphs for structured legal representation
• Strict adherence to Egyptian criminal law

KEY FEATURES:
-------------
✓ Strictly adherent to Egyptian law and Court of Cassation principles
✓ Multi-agent orchestration for specialized legal analysis
✓ Knowledge graph-based representation with full provenance tracking
✓ Procedural compliance auditing and evidence analysis
✓ Comprehensive citation and audit trail for all conclusions
✓ Explicit uncertainty flagging and reasoning transparency
✓ Human-in-the-loop design (assistive tool, not replacement)

CRITICAL DESIGN PRINCIPLES:
---------------------------
• NO external or non-Egyptian sources permitted
• NO free-text generation without citation
• NO judgments without complete reasoning
• EXPLICIT warnings for missing data or uncertainty
• DOUBT always interpreted in favor of defendant

================================================================================
                         3. CORE ARCHITECTURE
================================================================================

The system is organized into 5 distinct layers:

LAYER 1: INGESTION & PREPROCESSING
-----------------------------------
• Document intake (PDF, Text files)
• OCR and text extraction
• Data cleaning and normalization
• Fact/opinion separation
• Entity extraction

LAYER 2: KNOWLEDGE GRAPH (Neo4j)
---------------------------------
• Structured storage of legal knowledge
• Node types: Case, Fact, Charge, Evidence, Procedure, LegalArticle,
  CassationPrinciple, DefenseArgument, Judgment
• Relationship types: HAS_FACT, PROVES, NEGATES, APPLIES_TO, etc.
• Full provenance tracking for all data

LAYER 3: MULTI-AGENT ORCHESTRATOR
----------------------------------
• Orchestrator Agent (workflow control)
• Graph Builder Agent (document → graph)
• Procedural Auditor Agent (compliance checking)
• Legal Researcher Agent (law & precedent research)
• Evidence Analyst Agent (evidence evaluation)
• Defense Analysis Agent (defense argument analysis)
• Judge Agent (final reasoning and balancing)

LAYER 4: LLM REASONING LAYER
-----------------------------
• Constrained reasoning on structured data
• Citation requirements for all outputs
• Guardrails and validation
• Egyptian law focus only

LAYER 5: AUDIT / LOGGING / UI
------------------------------
• WORM (Write-Once-Read-Many) audit logs
• Complete provenance tracking
• User interface for case analysis
• Draft judgment generation

================================================================================
                        4. MULTI-AGENT SYSTEM
================================================================================

4.1 ORCHESTRATOR AGENT
----------------------
ROLE: Central coordinator and workflow manager

RESPONSIBILITIES:
• Receive case files (facts, investigations, evidence, reports, precedents)
• Distribute tasks to specialized agents
• Aggregate intermediate outputs
• Manage case state lifecycle
• Prevent agents from operating outside authorized context

GLOBAL CONTEXT MAINTAINED:
• Case identifier
• Current charges
• List of input documents
• List of disputed issues
• Snapshot hash of latest graph state

OUTPUTS:
• Task dispatch plan
• Approved context windows for each agent
• Aggregated final reports for Judge Agent

---

4.2 DATA INGESTION & GRAPH BUILDER AGENT
-----------------------------------------
ROLE: Convert unstructured documents to knowledge graph

CORE RULES:
• Transform text to structured graph representation
• NO interpretation, NO weighing, NO inference
• Extract facts ONLY (not opinions or characterizations)

INPUTS:
• PDF/Text files:
  - Referral orders (أوامر الإحالة)
  - Police reports (محاضر الشرطة)
  - Forensic reports (تقارير الطب الشرعي)
  - Prosecution investigations (تحقيقات النيابة)
  - Evidence items (أحراز)
  - Previous rulings (أحكام سابقة)

PROCESSING:
1. OCR and text chunking
2. Extract: facts, procedures, evidence, persons, charges
3. Convert to Neo4j nodes and relationships
4. Link each input with (docID, page, text span)

OUTPUTS:
• Executed Cypher queries
• Updated Neo4j graph database
• Provenance metadata for all extracted information

---

4.3 PROCEDURAL AUDITOR AGENT
-----------------------------
ROLE: Review procedural compliance

CORE RULES:
• Review procedures per Egyptian Criminal Procedure Code ONLY
• Apply Court of Cassation principles on nullity (relative vs. absolute)
• NO interpretation beyond statutory requirements

INPUTS:
• Arrest procedures (إجراءات القبض)
• Search procedures (إجراءات التفتيش)
• Interrogation procedures (إجراءات الاستجواب)
• Criminal Procedure Code articles
• Court of Cassation procedural principles

PROCESSING:
1. Match each procedure against legal requirements
2. Determine procedure validity
3. Identify procedural nullity (if applicable)
4. Assess impact on evidence (fruit of poisonous tree doctrine)

OUTPUT FORMAT (JSON):
{
  "procedure": "تفتيش (Search)",
  "status": "باطل (Null)",
  "nullity_type": "بطلان مطلق (Absolute Nullity)",
  "affected_evidence": ["E12", "E15"],
  "legal_basis": ["Article 45 - Procedure Code", "Cassation 12/3/1998"]
}

---

4.4 LEGAL RESEARCHER AGENT
---------------------------
ROLE: Disciplined legal research

CORE RULES:
• Legal research ONLY (no fact analysis, no evidence weighing)
• Automatic exclusion of all non-Egyptian sources

INPUTS:
• Charges from knowledge graph
• Keywords from case facts
• Extracted crime elements

PROCESSING (Graph RAG):
1. Vector search on legal articles and principles
2. Cypher expansion: articles → principles → precedents
3. Filter: Egyptian sources ONLY

OUTPUT FORMAT (JSON):
{
  "charge": "ضرب أفضى إلى موت (Battery resulting in death)",
  "statutes": ["Article 236 - Penal Code"],
  "principles": ["The criterion is proving criminal intent..."],
  "precedents": ["Cassation 5/6/2003", "Cassation 14/2/1999"]
}

---

4.5 EVIDENCE ANALYST AGENT
---------------------------
ROLE: Analyze probative value without final judgment

CORE RULES:
• Analyze evidence strength (NOT issue verdicts)
• Assess consistency, corroboration, contradictions

INPUTS:
• Evidence nodes, witness nodes, fact nodes from graph

PROCESSING:
LINKING:
• Physical evidence ↔ Facts
• Testimony ↔ Facts

EVALUATION:
• Internal consistency
• Corroboration across sources
• Contradictions

SEPARATION:
• Supporting evidence
• Contradicting evidence

OUTPUT FORMAT (JSON):
{
  "fact": "واقعة الاعتداء (Assault incident)",
  "supports": [
    {"evidence": "سلاح (Weapon)", "strength": 0.85},
    {"evidence": "شهادة X (Witness X)", "strength": 0.7}
  ],
  "contradictions": [
    {"evidence": "شهادة Y (Witness Y)", 
     "issue": "تناقض زمني (Temporal contradiction)"}
  ]
}

---

4.6 DEFENSE ANALYSIS AGENT
---------------------------
ROLE: Analyze defendant's defenses

CORE RULES:
• Analyze defenses (formal and substantive)
• NO fabrication of defenses not present in case

INPUTS:
• Graph subset: defenses, procedures, cassation principles
• Defense documents submitted

PROCESSING:
DEFENSE CLASSIFICATION:
• Procedural nullity (بطلان إجرائي)
• Insufficient evidence (قصور الأدلة)
• Internal contradictions (تناقض)
• Absence of criminal element (انتفاء ركن)

LINKING:
• Each defense → its legal basis

OUTPUT FORMAT (JSON):
{
  "formal_defenses": ["بطلان تفتيش (Nullity of search)"],
  "substantive_defenses": ["انتفاء القصد الجنائي (Absence of intent)"],
  "supporting_principles": ["Cassation 21/1/1982"]
}

---

4.7 JUDGE AGENT
---------------
ROLE: Final balancing and reasoning

CORE RULES:
• Balancing ONLY (no judgment without reasoning)
• NO weighing without legal basis
• EXPLICIT uncertainty declaration when applicable

INPUTS:
• Procedural auditor report
• Legal research findings
• Evidence analysis
• Defense arguments

REASONING PROCESS:
1. Present established facts ONLY
   (facts supported by procedurally sound evidence)

2. Present positions of parties:
   • Prosecution view (based on admitted evidence)
   • Defense (formal and substantive arguments)

3. Legal discussion:
   • Procedural validity
   • Evidence sufficiency
   • Statutory application

4. Rely EXCLUSIVELY on Court of Cassation principles

5. Reach suggested conclusion:
   • Conviction (إدانة)
   • Acquittal (براءة)
   • Lack of jurisdiction (عدم اختصاص)
   • Insufficient evidence (عدم كفاية الأدلة)

UNCERTAINTY HANDLING:
The Judge Agent includes EXPLICIT reservations when:
• Doubt exists
• Element is missing
• Evidence conflicts

HARDCODED PRINCIPLE:
"Doubt is ALWAYS interpreted in favor of the defendant"
(الشك يُفسَّر لمصلحة المتهم)

OUTPUTS:
• Reasoned draft judgment
• Explicit declaration of any deficiencies, doubts, or gaps

================================================================================
                      5. DATA PROCESSING PIPELINE
================================================================================

STAGE 1: DATA CLEANING & NORMALIZATION
---------------------------------------
• Deduplication
• Date unification (Hijri ↔ Gregorian to single format)
• Fact/opinion separation
  → Opinions, characterizations, inferences are ISOLATED
  → NOT fed to the graph

STAGE 2: KNOWLEDGE GRAPH CONVERSION
------------------------------------
GRAPH STRUCTURE:
• Each independent fact = Fact Node
• Each piece of evidence = Evidence Node
• Each procedure = Procedure Node
• Each legal article = LegalArticle Node
• Each cassation principle = CassationPrinciple Node

CRITICAL RESTRICTION:
✗ LLM does NOT see original text documents
✓ LLM sees ONLY: Nodes + Properties + Relationships + IDs

STAGE 3: GRAPH RAG USING Neo4j
-------------------------------
RETRIEVAL MECHANISMS:
• Cypher Queries (precise graph traversal)
• Subgraph Expansion (contextual information)
• Vector similarity search (semantic matching)

The LLM only sees query results, NEVER raw documents

STAGE 4: MERGING CYPHER AND REASONING
--------------------------------------
PROCESSING FLOW:
1. Query → 2. Legal Subgraph → 3. Structured Context (JSON) →
4. LLM Reasoning (constrained) → 5. Output (cited + reasoned)

STAGE 5: HALLUCINATION PREVENTION
----------------------------------
CRITICAL SAFEGUARDS:
• Disable free-text generation
• Citation REQUIRED for every inference
• Missing data → EXPLICIT WARNING + Suspension of judgment

WARNING TRIGGERS:
❗ Missing evidence
❗ Incomplete procedural record
❗ Conflicting information
❗ Gap in legal basis

================================================================================
                       6. KNOWLEDGE GRAPH SCHEMA
================================================================================

NODE TYPES:
-----------
• Case                  - Root node for criminal case
• Fact                  - Individual factual claim or event
• Charge                - Criminal charge against defendant
• LegalArticle          - Article from Egyptian Penal Code
• Evidence              - Physical evidence, testimony, documentation
• Procedure             - Procedural step (arrest, search, etc.)
• DefenseArgument       - Defense claim or argument
• CassationPrinciple    - Principle from Court of Cassation
• Judgment              - Final or intermediate ruling

RELATIONSHIP TYPES:
-------------------
(Case)-[:HAS_FACT]->(Fact)
    Case contains this fact

(Case)-[:HAS_CHARGE]->(Charge)
    Case includes this charge

(Charge)-[:APPLIES_TO]->(LegalArticle)
    Charge based on this legal article

(Evidence)-[:PROVES]->(Fact)
    Evidence supports/proves this fact

(Evidence)-[:NEGATES]->(Fact)
    Evidence contradicts this fact

(Procedure)-[:AFFECTS]->(Evidence)
    Procedure impacts evidence validity

(Procedure)-[:DEFECTIVE_BY]->(CassationPrinciple)
    Procedure violates this principle

(DefenseArgument)-[:REFUTES]->(Evidence)
    Defense challenges this evidence

(DefenseArgument)-[:BASED_ON]->(CassationPrinciple)
    Defense cites this principle

(Judgment)-[:BASED_ON]->(CassationPrinciple)
    Judgment relies on this principle

(Judgment)-[:EVALUATES]->(Evidence)
    Judgment weighs this evidence

================================================================================
                   7. AGENT COMMUNICATION PROTOCOL
================================================================================

COMMUNICATION RULES:
--------------------
All inter-agent communication follows STRICT protocol to ensure
traceability and prevent hallucination:

1. IMMUTABLE REFERENCES
   • All claims must reference Neo4j node IDs
   • No free-text summaries as sole references

2. STRUCTURED MESSAGES
   • JSON-serialized messages
   • Contains reference lists (node IDs)

3. PROVENANCE LOGGING
   • Complete audit trail in Neo4j
   • WORM (Write-Once-Read-Many) logs

MESSAGE FORMAT (JSON):
----------------------
{
  "task": "analyze_evidence",
  "sender": "evidence_analyst_agent",
  "receiver": "judge_agent",
  "references": [
    "node:E001",  // Evidence node
    "node:F042"   // Fact node
  ],
  "context_hash": "abc123...",
  "timestamp": "2024-02-01T10:30:00Z"
}

VERIFICATION FLOW:
------------------
1. Agent A queries Neo4j for relevant nodes
2. Neo4j returns node IDs
3. Agent A creates JSON message with node references
4. Agent A sends message to Agent B
5. Agent B verifies references against Neo4j
6. Both agents log operation to audit trail

================================================================================
                     8. HALLUCINATION PREVENTION
================================================================================

STRATEGY 1: DISABLE FREE-TEXT GENERATION
-----------------------------------------
• All outputs must be structured (JSON)
• All outputs must include source references
• No narrative without citations

STRATEGY 2: CITATION REQUIREMENTS
----------------------------------
RULE: Every inference MUST cite source node IDs

Example:
✓ CORRECT:
  "The defendant was at the scene [node:F012, node:W003]"

✗ INCORRECT:
  "The defendant was at the scene"

STRATEGY 3: MISSING DATA HANDLING
----------------------------------
Any missing data triggers:
1. EXPLICIT WARNING in output
2. SUSPENSION of judgment on affected elements

Example Warning:
"⚠️ WARNING: Forensic report on weapon (E015) is missing.
 Cannot determine cause of death with certainty.
 Judgment on murder charge SUSPENDED pending report."

STRATEGY 4: KNOWLEDGE BOUNDARY ENFORCEMENT
-------------------------------------------
• LLM cannot access information outside knowledge graph
• All queries pass through Graph RAG layer
• No general knowledge or analogies permitted

STRATEGY 5: VALIDATION CHECKPOINTS
-----------------------------------
Before final output, system validates:
✓ All facts cited to source documents
✓ All legal principles cited to statutes or precedents
✓ All procedures verified against legal requirements
✓ All conclusions supported by evidence chain

Audit Logs:

• Tamper-proof logging

Access Control:

• Multi-factor authentication
• Audit log for all access attempts

Data Residency:
• All data must remain within Egyptian jurisdiction
• No cloud services outside Egypt
• Local deployment preferred

Accuracy Targets:
• Procedural compliance detection: 95%+
